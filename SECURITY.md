# سياسة الأمان - RMDATA

## نطاق السياسة

هذا المشروع نظام داخلي لإدارة بيانات شركة الرداء الموحد. يحتوي على بيانات تشغيلية وحساسة مثل بيانات الموظفين، المستندات، المستخدمين، الصلاحيات، والاتصال بقاعدة البيانات.

## الإبلاغ عن مشكلة أمنية

إذا تم اكتشاف مشكلة أمنية، لا تفتح Issue عامة ولا تكتب التفاصيل في commit أو pull request عام.

يجب الإبلاغ مباشرة لصاحب المشروع أو المسؤول التقني عبر قناة خاصة، مع توضيح:

- وصف مختصر للمشكلة.
- الخطوات التي تؤدي للمشكلة.
- الملفات أو المسارات المتأثرة إن وجدت.
- درجة الخطورة المتوقعة.

## معلومات لا يجب رفعها إلى Git

يمنع رفع أي من التالي إلى المستودع:

- ملفات `.env` الحقيقية.
- كلمات مرور قاعدة البيانات.
- مفاتيح JWT أو API secrets.
- نسخ قواعد البيانات `.db`.
- ملفات backup.
- ملفات المستخدمين والمستندات الحقيقية.
- ملفات build أو release إذا لم تكن مطلوبة صراحة.

## قواعد التطوير الآمن

- لا يتم تغيير قاعدة البيانات بدون backup وmigration واضح.
- لا يتم الاعتماد على صلاحيات الواجهة فقط؛ يجب تطبيق الصلاحيات في backend.
- أي endpoint حساس يجب أن يستخدم authentication وpermission checks.
- لا يتم تسجيل كلمات المرور أو التوكنات في logs.
- أي تغيير في صلاحيات المستخدمين يجب أن يترك أثراً في audit/activity logs.
- لا يتم تعطيل TypeScript strictness لإخفاء أخطاء أمنية أو منطقية.

## الملفات الحساسة تقنياً

يجب مراجعة هذه المناطق بحذر عند التعديل:

- `electron/remote-api-utils.ts`
- `electron/ipc/auth-ipc.ts`
- `electron/ipc/settings-ipc.ts`
- `server/dev-api-server.js`
- `server/permissions-resolver.js`
- `server/permission-middleware.js`
- `api-gateway-php/api/index.php`
- `api-gateway-php/src/auth.php`
- `api-gateway-php/src/db.php`
- `src/permissions/permissionCatalogV2.ts`

## حالة المشروع الحالية

المشروع في مرحلة تجهيز للانتقال إلى V2. الأولوية الأمنية الحالية:

1. تثبيت Git baseline.
2. أخذ backups.
3. إصلاح TypeScript.
4. تقليل الاعتماد على raw SQL عبر `/api/db/query`.
5. توحيد Node API كمسار أساسي للوضع البعيد.
