# تصميم وحدة الإعدادات — النسخة الأولى (V1)

## 1. Executive Summary

هذا المستند يقدّم مقترح تصميم **وحدة الإعدادات (Settings)** لمنصة إدارة العمليات الداخلية (ERP) دون تنفيذ أي كود أو تغيير سلوك التشغيل الحالي. الهدف هو الحصول على هيكل واضح وعملي للإعدادات، مع دعم إدارة المستخدمين والصلاحيات وتخزين كلمات المرور بشكل آمن (BCrypt)، وإعدادات التنبيهات والنسخ الاحتياطي، مع تمييز واضح بين ما هو أساسي في V1 وما يُؤجّل إلى V2.

**المبادئ التوجيهية:**
- عملي لنظام شركة داخلي (لا تعقيد على مستوى enterprise إلا عند الحاجة).
- بسيط، آمن، قابل للصيانة والتوسع لاحقاً (مستخدمون متعددون، أجهزة متعددة، تطبيق جوال).
- جاهز لـ V1 مع إمكانية تأجيل أجزاء إلى V2.

---

## 2. Recommended Settings Module Structure

يُقترح تنظيم صفحة الإعدادات كصفحة رئيسية تحتوي على **أقسام (sections)**، كل قسم قد يكون صفحة فرعية أو تبويبات داخل نفس الصفحة حسب الحجم.

| القسم | الوصف | V1 / V2 | الحالة |
|-------|--------|---------|--------|
| **General / هوية البرنامج** | اسم الشركة، الشعار | V1 أساسي | ✅ منفّذ |
| **Language / اللغة** | لغة الواجهة (عربي / إنجليزي) — خيارات ثنائية اللغة | V1 أساسي | ✅ منفّذ |
| **Users / المستخدمون** | إدارة حسابات المستخدمين (إضافة، تعديل، تفعيل/إيقاف) | V1 أساسي | ✅ منفّذ |
| **Roles & Permissions** | الأدوار والصلاحيات المرتبطة بها | V1 أساسي | ✅ منفّذ (عرض فقط) |
| **Notifications** | إعدادات الإشعارات العامة وتنبيهات الانتهاء (أخضر/أصفر/أحمر) | V1 أساسي | ✅ منفّذ |
| **Database / قاعدة البيانات** | نوع المحرك، حالة الاتصال، إحصائيات الجداول | V1 أساسي | ✅ **جديد** |
| **Backup & Restore** | نسخ احتياطي يدوي، استعادة | V1 أساسي | ✅ منفّذ |
| **About / حول النظام** | الإصدار، لوجو، تحقق تحديثات، ترخيص، نسخ معلومات | V1 أساسي | ✅ **جديد** |
| **Documents** | مسارات المستندات الافتراضية، أحجام التحميل | V2 | ⬜ مؤجل |
| **System Behavior** | سلوكيات عامة (تأكيد حذف، حفظ تلقائي) | V2 | ⬜ مؤجل |

**هيكل التنقل المقترح:**
- مسار واحد: `/dashboard/settings` مع استعلام أو مسار فرعي للقسم النشط، مثلاً:
  - `/dashboard/settings` → عرض لوحة أقسام الإعدادات (بطاقات أو قائمة).
  - `/dashboard/settings/general` أو `?section=general`
  - `/dashboard/settings/users`
  - `/dashboard/settings/roles`
  - `/dashboard/settings/notifications`
  - `/dashboard/settings/database`
  - `/dashboard/settings/backup`
  - `/dashboard/settings/about`

لا يُنفّذ تغيير في الـ routing الحالي في هذه المرحلة؛ الهيكل أعلاه للتصميم فقط.

## 3. Version 1 Essential Sections (Must-Have for V1)

لنسخة V1، الأقسام **الإلزامية** هي:

1. **General (هوية البرنامج)**  
   - اسم النظام/الشركة، شعار (اختياري)، اللغة الافتراضية، المنطقة الزمنية، تنسيق التاريخ.  
   - ضروري لتمييز البرنامج وللتوحيد لاحقاً مع التقارير والإشعارات.

2. **Language**  
   - لغة الواجهة (عربي / إنجليزي) كحد أدنى.  
   - يمكن تخزينها في جدول إعدادات عامة أو في `localStorage` مع مفتاح في الإعدادات.

3. **Users**  
   - إنشاء/تعديل/تعطيل المستخدمين، ربطهم بدور، وعدم تخزين كلمة المرور بنص واضح (راجع البند 7).

4. **Roles & Permissions**  
   - نموذج أدوار ثابت أو قليل (مثل Admin, Manager, Staff, Viewer) مع مصفوفة صلاحيات واضحة (عرض/تعديل/حذف لكل وحدة).  
   - ضروري لأي تحكم بالوصول قبل تفعيل المستخدمين الحقيقيين.

5. **Notifications**  
   - تفعيل/تعطيل الإشعارات العامة، وتنبيهات الانتهاء.  
   - التحكم في إظهار التنبيه الأخضر (ساري) والأصفر (قريب من الانتهاء) مع **الإبقاء على الأحمر (منتهي) دائماً ظاهراً**.

6. **Expiry Alerts**  
   - عدد الأيام قبل الانتهاء لاعتبار العنصر "تحذير" (مثلاً 30 يوم).  
   - يمكن دمج هذا مع "Notifications" في واجهة واحدة إذا رُئي أبسط.

7. **Backup & Restore**  
   - نسخ احتياطي يدوي لملف قاعدة البيانات (وربما مجلد المستندات).  
   - استعادة من ملف نسخة.  
   - تصدير/استيراد اختياري (مثلاً JSON/CSV) لنسخ محدودة إن لزم.

## 5. Users Management Proposal

### 5.1 الحقول المقترحة للمستخدم (جدول `users` أو ما يعادله)

| الحقل | النوع | ملاحظات |
|--------|--------|---------|
| `id` | INTEGER PK | معرف فريد |
| `username` | VARCHAR(50) UNIQUE | اسم الدخول (إجباري، لا مسافات) |
| `passwordHash` | VARCHAR(255) | **فقط hash (BCrypt)** — لا يُخزّن أبداً نص واضح |
| `fullName` | VARCHAR(200) | الاسم الكامل للعرض |
| `email` | VARCHAR(255) | اختياري للاستعادة والإشعارات لاحقاً |
| `roleId` | INTEGER FK → roles | ربط بدور واحد |
| `isActive` | BOOLEAN (0/1) | تفعيل/إيقاف الحساب |
| `branchId` | INTEGER NULL FK → branches | اختياري: تقييد وصول المستخدم لفرع معيّن (لـ V2 أو لاحقاً) |
| `lastLoginAt` | DATETIME NULL | آخر تسجيل دخول (للمراجعة فقط) |
| `createdAt` | DATETIME | |
| `updatedAt` | DATETIME | |

**لا يُضاف أبداً:** حقل `password` بنص واضح، ولا أي حقل يحفظ كلمة المرور الأصلية.

### 5.2 إنشاء المستخدمين

- **من لديه صلاحية:** فقط أدوار من نوع "Admin" أو من يملك صلاحية `users.create`.
- **آلية مقترحة:** نموذج "إضافة مستخدم" يطلب: `username`, `fullName`, `email` (اختياري), `roleId`, وكلمة مرور أولية (مرتين).  
- **عند الحفظ:**  
  - حساب hash لكلمة المرور باستخدام BCrypt (في عملية Main أو في API آمن).  
  - حفظ `passwordHash` فقط في قاعدة البيانات.  
  - عدم طباعة أو تسجيل كلمة المرور أو الـ hash في الـ logs.

### 5.3 التفعيل / التعطيل

- حقل `isActive`: `1` = مفعّل، `0` = معطّل.  
- المستخدم المعطّل لا يستطيع تسجيل الدخول؛ يظهر في قائمة المستخدمين مع تمييز (مثلاً لون أو أيقونة).  
- لا حذف فعلي للحسابات في V1 (soft delete عبر `isActive` كافٍ).

### 5.4 استعادة كلمة المرور

- **إعادة تعيين من قبل Admin (V1):**  
  - مسار من نوع "إعادة تعيين كلمة المرور" لمستخدم معيّن: يدخل المسؤول كلمة مرور جديدة (مرتين)، يُحسب الـ BCrypt hash ويُحدَّث عمود `passwordHash` فقط.  
- **نسيان كلمة المرور (رابط بريد / OTP):** يُؤجّل إلى V2 (يتطلب إعداد بريد أو خدمة إشعارات).

### 5.5 الحد الأدنى للأمان

- كلمات المرور: **فقط BCrypt**، لا تخزين ولا عرض ولا تسجيل لنص واضح.  
- عدم تمرير كلمة المرور في الـ URL أو في الـ query params.  
- استخدام HTTPS في أي اتصال شبكي مستقبلي.  
- تقييد واجهات إدارة المستخدمين والصلاحيات بصلاحيات واضحة (راجع البند 6).

## 6. Roles and Permissions Proposal

### 6.1 نموذج الأدوار المقترح (عملي لشركة داخلية)

| الدور | الوصف | العرض | التعديل | الحذف/الأرشفة | ملاحظات |
|-------|--------|--------|---------|----------------|----------|
| **Admin** | مدير النظام | كل الوحدات | كل الوحدات | نعم | إدارة المستخدمين والأدوار والإعدادات |
| **Manager** | مدير عمليات | كل الوحدات | الموظفون، الأفرع، السكن، المركبات، الهواتف، أصحاب العمل (حسب الصلاحيات التفصيلية) | حسب الصلاحية | لا يصل لإعدادات المستخدمين/الأدوار عادة |
| **Staff** | موظف إدخال | الوحدات المصرح بها | نفس الوحدات بحدود (مثلاً لا حذف) | لا أو محدود | لا يصل للإعدادات الحساسة |
| **Viewer** | عرض فقط | كل أو جزء من الوحدات | لا | لا | للتقارير والمراجعة فقط |

يمكن اختصار V1 إلى: **Admin** و **Manager** و **Staff** و **Viewer** مع مصفوفة صلاحيات ثابتة في الكود أو في جدول `permissions` (انظر أدناه).

### 6.2 ما يمكن لكل دور رؤيته/تعديله (مثال)

- **Admin:** الإعدادات، المستخدمون، الأدوار، كل الأقسام (أفرع، موظفون، سكن، مركبات، هواتف، أصحاب عمل، ضرائب، مستندات، أرشيف، سجل النظام).  
- **Manager:** كل البيانات التشغيلية؛ يمكن منعه من "المستخدمون والأدوار" و"الإعدادات العامة الحساسة" (مثل النسخ الاحتياطي).  
- **Staff:** عرض وتعديل حسب الوحدات المفعّلة له؛ بدون حذف أو أرشفة إن لم تُمنح صراحة.  
- **Viewer:** عرض فقط للوحدات المحددة؛ لا أزرار تعديل/حذف.

### 6.3 ما لا يُمنح لأي دور عادي

- الوصول المباشر إلى ملف قاعدة البيانات أو نظام الملفات.  
- تغيير صلاحيات الأدوار أو إنشاء أدوار جديدة (مخصص لـ Admin فقط في V1).  
- عرض أو تعديل حقول مثل `passwordHash` أو مفاتيح التشفير إن وُجدت.

### 6.4 تخزين الصلاحيات (منظم وقابل للتوسع)

- **جدول `roles`:**  
  - `id`, `name` (مثل "Admin", "Manager"), `description`, `isSystem` (لا يحذف إذا كان نظامياً).

- **جدول `permissions`:**  
  - صلاحيات على مستوى **الوحدة + الإجراء**، مثلاً:  
    - `module`: 'employees' | 'branches' | 'housing' | 'vehicles' | 'employers' | 'phones' | 'entities' | 'documents' | 'settings' | 'users' | 'logs' | ...  
    - `action`: 'view' | 'create' | 'edit' | 'delete' | 'archive' | 'manage' (للمستخدمين/الأدوار).  

- **جدول `role_permissions`:**  
  - `roleId`, `permissionId` (أو دمج كـ `roleId`, `module`, `action`).  

- **التحقق في التطبيق:**  
  - دالة مركزية من نوع `can(user, 'employees', 'edit')` تُستدعى قبل عرض أزرار التعديل أو قبل تنفيذ الإجراء.  
  - عدم نشر صلاحيات كنص مبعثر في الكود؛ استخدام ثوابت (مثلاً `PERMISSIONS.EMPLOYEES_EDIT`) أو مفاتيح من قاعدة البيانات.

هذا يسمح لاحقاً بـ "أدوار مخصصة" في V2 دون إعادة تصميم كامل.

## 7. BCrypt Password Handling Recommendations

- **التخزين:**  
  - عمود واحد فقط: `passwordHash` (VARCHAR 255).  
  - استخدام BCrypt مع عامل تكلفة مناسب (مثلاً 10–12).  
  - عدم تخزين الملح بشكل منفصل (BCrypt يضمّنه في الـ hash).

- **عند التسجيل/التحديث:**  
  - استلام كلمة المرور فقط من واجهة آمنة (مثلاً حقل type="password").  
  - حساب الـ hash في عملية موثوقة (Main في Electron أو backend إن وُجد).  
  - عدم إرسال كلمة المرور الأصلية إلى الـ renderer أو تسجيلها في الـ logs.

- **عند تسجيل الدخول:**  
  - مقارنة المدخل مع `passwordHash` عبر دالة مثل `bcrypt.compare(plainPassword, storedHash)`.  
  - عدم طباعة أو تسجيل نتيجة المقارنة أو أي جزء من كلمة المرور.

- **في الكود والتوثيق:**  
  - التأكيد في تعليقات ووثائق المشروع أن كلمات المرور لا تُخزّن ولا تُعرض ولا تُسجّل كنص واضح.  
  - مراجعة أي نقطة جديدة (استعادة كلمة المرور، تغيير كلمة المرور) للتأكد من عدم تسريب النص الواضح.

## 8. Notification Settings Proposal

### 8.1 إعدادات عامة للإشعارات

- **تفعيل/تعطيل إشعارات النظام:** خيار عام (مثلاً "تفعيل الإشعارات") يتحكم في إنشاء إشعارات جديدة في جدول `notifications` (انتهاء صلاحيات، استحقاقات، إلخ).  
- **عرض الإشعارات في الواجهة:** إبقاء الجرس/القائمة الحالية مع إمكانية إضافة "لا تُظهر الإشعارات في الشريط العلوي" كخيار للمستخدم (يُخزَن في تفضيلات المستخدم أو في الإعدادات العامة).

### 8.2 تنبيهات الانتهاء (Expiry)

- **عدد أيام التحذير (أصفر):** قيمة افتراضية 30 يوماً؛ قابلة للتعديل من الإعدادات (مثلاً 7، 14، 30، 60).  
- **إظهار/إخفاء حسب اللون:**  
  - **أخضر (ساري):** خيار "إظهار الحالة السارية" — إن أُلغي، لا يُعرض سطر "ساري" حيث لا حاجة للمستخدم.  
  - **أصفر (قريب من الانتهاء):** خيار "إظهار تحذير قرب الانتهاء" — إن أُلغي، لا يُعرض التحذير الأصفر.  
  - **أحمر (منتهي):** **دائماً ظاهر** ولا يوجد خيار لإخفائه؛ لأسباب تشغيلية وأمنية.

### 8.3 التحكم البصري

- يمكن تمثيل ذلك في واجهة "الإشعارات وتنبيهات الانتهاء" بأزرار أو مفاتيح:  
  - "أيام التحذير قبل الانتهاء": حقل رقمي.  
  - "إظهار التنبيه الأخضر (ساري)"، "إظهار التنبيه الأصفر (قريب من الانتهاء)" — مع توضيح أن الأحمر لا يُخفى.

## 9. Backup Settings Proposal

### 9.1 نسخ احتياطي يدوي (V1)

- زر "إنشاء نسخة احتياطية الآن":  
  - نسخ ملف قاعدة البيانات (SQLite) إلى مجلد يختاره المستخدم أو مجلد افتراضي (مثل `Backups` ضمن مجلد التطبيق) مع اسم يتضمن التاريخ والوقت.  
  - اختياري: أرشفة مجلد المستندات الرئيسي في نفس العملية أو في خطوة منفصلة.

### 9.2 الاستعادة (V1)

- زر "استعادة من نسخة":  
  - اختيار ملف نسخة (`.db` أو ملف معرّف للتطبيق).  
  - تحذير واضح بأن الاستعادة ستستبدل البيانات الحالية.  
  - إغلاق أي اتصال بقاعدة البيانات الحالية، استبدال الملف، إعادة فتح الاتصال وإعادة تشغيل واجهة التطبيق إن لزم.

### 9.3 تصدير / استيراد (V1 اختياري)

- **تصدير:** تصدير جداول محددة (مثلاً الموظفون، الأفرع) إلى JSON أو CSV للنسخ أو المراجعة.  
- **استيراد:** استيراد من نفس التنسيق مع التحقق من الصيغة وتجنب التكرار (مثلاً بالمعرف أو بالرمز).  
- يمكن تأجيل "استيراد كامل" إلى V2 والاكتفاء بتصدير بسيط في V1.

### 9.4 نسخ تلقائي (V2)

- جدولة نسخ دورية (يومية/أسبوعية) وحفظها في مجلد معيّن مع سياسة احتفاظ (مثلاً آخر 7 نسخ).  
- لا تنفيذ في V1 لتبسيط المتطلبات.

## 10. Suggested Database Tables / Entities

### 10.1 جداول جديدة مقترحة (للمرجع فقط — لا تنفيذ في هذه المرحلة)

**`settings` (مفتاح–قيمة أو صف واحد)**  
- `id`, `key` (VARCHAR unique), `value` (TEXT/JSON), `updatedAt`  
- أمثلة مفاتيح: `companyName`, `logoPath`, `defaultLanguage`, `timezone`, `dateFormat`, `expiryWarningDays`, `showGreenExpiry`, `showYellowExpiry`, `notificationsEnabled`.

**`users`**  
- كما في البند 5.1 مع `passwordHash` وبدون أي حقل كلمة مرور نصية.

**`roles`**  
- `id`, `name`, `description`, `isSystem`.

**`permissions`**  
- `id`, `module`, `action`, `labelKey` (للعرض في الواجهة).

**`role_permissions`**  
- `roleId`, `permissionId` (أو `module`, `action` مباشرة).

**`user_sessions` (اختياري لـ V1)**  
- لتسجيل جلسات الدخول أو إبطالها لاحقاً؛ يمكن تأجيله.

لا تغيير على جداول `activity_logs` أو `notifications` في التصميم الحالي إلا إذا رُئي ربط إعدادات بالإشعارات (مثل تعطيل إنشاء إشعارات من الإعدادات).

    - المستخدمون  
    - الأدوار والصلاحيات  
    - الإشعارات وتنبيهات الانتهاء  
    - النسخ الاحتياطي والاستعادة  

- **كل قسم:**  
  - عنوان فرعي واضح.  
  - نموذج أو خيارات مع أزرار "حفظ" حيث يلزم.  
  - عدم حفظ كلمات المرور في الحقول؛ استخدام حقول من نوع "كلمة مرور" مع إخفاء الإدخال.

- **التنقل:**  
  - إما تبويبات أفقية/عمودية داخل الصفحة، أو مسارات فرعية كما في البند 2.  
  - الإعدادات الحساسة (مستخدمون، أدوار، نسخ احتياطي) يفضّل أن تظهر فقط للمستخدمين الذين يملكون الصلاحية المناسبة.

- **الاتساق:**  
  - نفس أسلوب التصميم المستخدم في باقي التطبيق (مثل صفحة الموظفين/الأفرع): ألوان، خطوط، تخطيط RTL.

## 12. Implementation Roadmap in Phases

### المرحلة 0 — التحضير (قبل كتابة كود الإعدادات)
- توثيق التصميم (هذا المستند) والاتفاق عليه.  
- إعداد بيئة BCrypt في المشروع (مكتبة، دوال آمنة في Main).  
- تعريف جداول `settings`, `users`, `roles`, `permissions`, `role_permissions` في الـ migrations دون ربطها بعد بواجهات حية.

### المرحلة 1 — الأساس
- تنفيذ صفحة الإعدادات الرئيسية مع القائمة/البطاقات.  
- تنفيذ قسم "هوية البرنامج / عام" و"اللغة" مع قراءة/كتابة من جدول `settings` (أو مصدر موحد).  
- ربط واجهة اللغة (عربي/إنجليزي) إن وُجدت مع التطبيق.

### المرحلة 2 — المستخدمون والأدوار
- تنفيذ جداول الأدوار والصلاحيات وربطها.  
- تنفيذ إدارة المستخدمين (إضافة، تعديل، تفعيل/إيقاف) مع تخزين كلمة المرور بـ BCrypt فقط.  
- تنفيذ تسجيل الدخول الحقيقي (مقارنة BCrypt) وربط `authStore` بالمستخدم من قاعدة البيانات.  
- إعادة تعيين كلمة المرور من قبل Admin.

### المرحلة 3 — الصلاحيات في الواجهة
- دالة تحقق مركزية من الصلاحيات.  
- إخفاء/إظهار أزرار وقوائم حسب دور المستخدم (عرض، تعديل، حذف، إعدادات).  
- حماية مسارات الإعدادات والمستخدمين والأدوار بصلاحية واضحة.

### المرحلة 4 — الإشعارات وتنبيهات الانتهاء
- واجهة إعدادات الإشعارات وتنبيهات الانتهاء (أيام التحذير، إظهار أخضر/أصفر).  
- ربط منطق الواجهة (أخضر/أصفر/أحمر) بقيم الإعدادات مع الإبقاء على الأحمر دائماً ظاهراً.

### المرحلة 5 — النسخ الاحتياطي والاستعادة
- نسخ احتياطي يدوي لملف قاعدة البيانات.  
- استعادة من ملف.  
- (اختياري) تصدير بسيط لجدول أو أكثر.

## Recommended Implementation Order for the Settings Module

الترتيب المقترح للتنفيذ:

1. **الإعدادات العامة وهوية البرنامج + اللغة**  
   - لا تعتمد على مستخدمين/صلاحيات معقدة؛ تعطي قيمة فورية وتمهد لواجهة الإعدادات.

2. **جداول المستخدمين والأدوار والصلاحيات + BCrypt**  
   - أساس أمان النظام؛ يجب أن يكون جاهزاً قبل تفعيل تسجيل الدخول الحقيقي.

3. **إدارة المستخدمين (واجهة) + تسجيل الدخول بـ BCrypt**  
   - ربط الواجهة بقاعدة المستخدمين وجعل الدخول يعتمد على كلمة المرور فقط (hash).

4. **ربط الصلاحيات بالواجهة والمسارات**  
   - بعد وجود أدوار وصلاحيات، تطبيق التحقق في القوائم والأزرار والمسارات.

5. **إعدادات الإشعارات وتنبيهات الانتهاء**  
   - تعتمد على وجود "إعدادات" عامة وربما صلاحيات؛ يمكن تنفيذها بعد استقرار المستخدمين.

6. **النسخ الاحتياطي والاستعادة**  
   - مستقل نسبياً؛ يمكن تنفيذه متوازياً أو بعد الإشعارات، مع تقييد وصوله بصلاحية Admin.

هذا الترتيب يحافظ على أمان النظام (كلمات مرور وصلاحيات أولاً) ويجعل كل مرحلة قابلة للاختبار دون تعقيد زائد.
