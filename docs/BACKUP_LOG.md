# سجل النسخ الاحتياطي - RMDATA

آخر تحديث: 2026-05-13

## الغرض

هذا الملف يسجل حالة النسخ الاحتياطي قبل بدء تنظيف V2. الهدف أن نعرف أين توجد النسخ وكيف نرجع لها إذا صار خطأ أثناء التطوير أو التحديث.

## الحالة الحالية

حسب تأكيد صاحب المشروع بتاريخ 2026-05-13:

- النسخ الاحتياطي على السيرفر يعمل حالياً.
- التحديثات مرفوعة ومضبوطة لتذهب إلى السيرفر.
- النسخة المثبتة الحالية لن يتم لمسها مباشرة أثناء التطوير.
- التطوير سيتم داخل Git، ثم يتم عمل build جديد فقط بعد الاختبار.

## مصادر النسخ الاحتياطي المطلوبة

| المصدر | الحالة | الملاحظات |
|---|---|---|
| MySQL/MariaDB على VPS | مؤكد من المالك أنه موجود/يعمل | يفضل لاحقاً تسجيل أمر النسخ والمسار الفعلي |
| ملفات المستندات والصور على السيرفر | مؤكد من المالك أن النسخ الاحتياطي يعمل | المسار المؤكد: `/var/www/api.rmdata.tech/storage` |
| ملفات التحديثات على السيرفر | مؤكد من المالك أنها تذهب للسيرفر | المسار المؤكد: `/var/www/api.rmdata.tech/html/updates` |
| SQLite محلي | غير مؤكد كاحتياج إنتاجي | إذا توجد أجهزة تعمل local فقط، يجب أخذ نسخة من ملف `.db` |
| كود المشروع | تم رفعه إلى GitHub | `https://github.com/khatib96/rmdata.git` |

## قاعدة العمل قبل أي Migration

قبل أي تغيير في قاعدة البيانات:

1. تأكيد وجود backup حديث.
2. تسجيل تاريخ ووقت النسخة هنا.
3. تنفيذ migration.
4. اختبار التطبيق.
5. أخذ backup جديد بعد نجاح migration إذا كان التغيير مهماً.

## سجل النسخ

### 2026-05-13 - baseline قبل V2

الحالة:

- تم تثبيت Git baseline.
- تم رفع المشروع إلى GitHub.
- تم تأكيد أن نسخ السيرفر الاحتياطية تعمل من طرف صاحب المشروع.
- لم يتم تنفيذ migration جديد في هذه الجلسة.
- تم فحص السيرفر قراءة فقط وتوثيق المسارات الأساسية:
  - التطبيق: `/var/www/api.rmdata.tech/current`
  - الملفات: `/var/www/api.rmdata.tech/storage`
  - المستندات: `/var/www/api.rmdata.tech/storage/documents`
  - الصور: `/var/www/api.rmdata.tech/storage/images`
  - PM2 API: `rmdata-node-api`

الملاحظات:

- تم توثيق مسارات السيرفر الفعلية في `CURRENT_STATE_v1.4.1.md`.
- مسار التحديثات المؤكد: `/var/www/api.rmdata.tech/html/updates`.
- لا يتم رفع أي تحديث للمستخدمين قبل build واختبار.
