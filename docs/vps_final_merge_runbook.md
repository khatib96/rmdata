# Runbook نهائي: دمج بيانات الاستضافة المشتركة مع VPS (قاعدة + صور + مستندات)

## الهدف

تنفيذ دمج نهائي آمن عندما:
- يوجد بيانات جديدة على الاستضافة المشتركة.
- ويوجد أيضا بيانات جديدة على VPS.
- والمطلوب الحفاظ على كل التعديلات من الطرفين.

---

## قبل البدء (إجباري)

- [ ] تحديد نافذة صيانة (30-90 دقيقة حسب حجم البيانات).
- [ ] إيقاف الإدخال على الطرفين (Read Only مؤقتا أو إيقاف واجهة الإدخال).
- [ ] أخذ نسخة احتياطية كاملة من:
  - [ ] قاعدة الاستضافة المشتركة.
  - [ ] قاعدة VPS.
  - [ ] مجلدات الصور والمستندات على VPS.

> لا تبدأ الدمج بدون Backup للطرفين.

---

## المرحلة 1: تجهيز قواعد الدمج على VPS

### 1.1 إنشاء قاعدة مؤقتة لاستقبال نسخة القديم
- أنشئ قاعدة مؤقتة على VPS باسم واضح، مثال:
  - `rmdata_old_snapshot`

### 1.2 استيراد نسخة الاستضافة المشتركة داخل القاعدة المؤقتة
- استورد ملف SQL الكامل القادم من الاستضافة المشتركة إلى:
  - `rmdata_old_snapshot`

### 1.3 تأكيد الاستيراد
- تأكد أن الجداول موجودة وأن عدد الصفوف منطقي.

---

## المرحلة 2: دمج قاعدة البيانات (Table-by-Table)

> القاعدة الأساسية النهائية: `rmdata_db`  
> المصدر القديم: `rmdata_old_snapshot`

## 2.1 الجداول السجلية (Logs/History)

أمثلة: `activity_logs`, `status_history`, ...  
الاستراتيجية:
- `INSERT IGNORE` من القديم إلى الأساسي.
- السبب: هذه جداول تراكمية، نضيف ما ينقص بدون استبدال.

Checklist:
- [ ] دمج `activity_logs`
- [ ] دمج `status_history`
- [ ] دمج أي جدول سجل مشابه

## 2.2 الجداول الرئيسية (Entities)

أمثلة: `employees`, `branches`, `users`, `documents`, `vehicles`, ...  
الاستراتيجية:
- Upsert (إدخال عند عدم وجود السجل + تحديث عند وجوده).
- عند التعارض، يعتمد الحسم على `updatedAt` (الأحدث يفوز).

Checklist:
- [ ] دمج `employees` بـ Upsert
- [ ] دمج `branches` بـ Upsert
- [ ] دمج `users` بـ Upsert
- [ ] دمج `documents` بـ Upsert
- [ ] دمج بقية الجداول الأساسية بنفس النمط

## 2.3 جداول الربط (Pivot/Mapping)

أمثلة: `role_permissions`, `tax_entity_branches`, ...  
الاستراتيجية:
- `INSERT IGNORE` غالبا يكفي إذا المفاتيح المركبة معرفة بشكل صحيح.

Checklist:
- [ ] دمج جداول الربط

## المرحلة 3: دمج الصور والمستندات

## 3.1 تحديد مسارات الملفات الفعلية
- المسار المعتمد فعليا للرفع/القراءة في نسخة VPS:
  - `api-gateway-php/storage` (Storage Root موحد)
- لا تعتمد على مجلدات legacy مثل:
  - `/var/www/.../uploads`
  - `/var/www/.../images`
- الصور والمستندات يجب أن تكون داخل `storage` بمسارات نسبية منظمة مثل:
  - `images/branches/...`
  - `images/employees/...`
  - `images/employers/...`
  - `Branches/<branchId>/...`
  - `Employees/<employeeId>/...`
  - `Employers/<employerId>/...`

## 3.2 نسخ الملفات من القديم إلى VPS بدون حذف الموجود

الهدف:
- إضافة الملفات الناقصة من القديم.
- عدم حذف ملفات VPS.
- عدم استبدال الملف إذا نسخة VPS أحدث.

الاستراتيجية الموصى بها:
- `rsync -av --update` (أو أي أداة مكافئة بنفس المنطق).

Checklist:
- [ ] مزامنة مجلد الصور
- [ ] مزامنة مجلدات المستندات
- [ ] التأكد من الصلاحيات بعد النسخ

## 3.3 تحقق تطابق المسارات
- تأكد أن القيم المخزنة في DB (`imagePath` / `filePath`) تشير لمسارات موجودة فعليا على VPS.
- يجب أن تكون القيم في DB مسارات نسبية فقط (بدون مسارات نظام تشغيل كاملة).

## 3.4 ضبط حدود رفع الملفات (منع خطأ 413)

سبب شائع بعد النقل إلى VPS:  
`413 Request Entity Too Large` من `nginx` أثناء `POST /api/files/upload`.

### Nginx
- اضبط `client_max_body_size` بقيمة أعلى من أكبر ملف متوقع (مثال 20M أو 25M):

```nginx
server {
    client_max_body_size 25M;
    # ...
    location /api/ {
        client_max_body_size 25M;
        proxy_pass http://php_backend;
    }
}
```

### PHP
- اجعل القيم متناسقة مع Nginx:
  - `upload_max_filesize = 25M`
  - `post_max_size = 25M`
- في API gateway، تأكد أن `MAX_UPLOAD_MB` في `.env` متوافق (مثال: `MAX_UPLOAD_MB=20`).

### بعد التعديل
- نفّذ إعادة تحميل للخدمات:
  - `nginx -t && systemctl reload nginx`
  - إعادة تحميل PHP-FPM (حسب الإصدار).
- اختبر رفع صورة جديدة من التطبيق (وليس تعديل صورة قديمة فقط).

## 3.5 تحقق VPS لمسارات وصلاحيات التخزين

- تأكد أن مجلد التخزين موجود:
  - `ls -lah /var/www/.../api-gateway-php/storage`
- تأكد من المالك/المجموعة لمستخدم خدمة الويب (`www-data` غالبا):
  - `chown -R www-data:www-data /var/www/.../api-gateway-php/storage`
- الصلاحيات الموصى بها:
  - `find /var/www/.../api-gateway-php/storage -type d -exec chmod 775 {} \\;`
  - `find /var/www/.../api-gateway-php/storage -type f -exec chmod 664 {} \\;`
- اختبار API مباشر:
  - `GET /api/health` يجب أن يرجع `ok=true`.
  - رفع ملف صورة اختبار ثم التحقق من وجوده داخل `storage/images/...`.

## المرحلة 4: التحقق النهائي قبل إعادة التشغيل

## 4.1 تحقق كمي (Counts)
- قارن عدد الصفوف لكل جدول رئيسي بين:
  - `rmdata_db`
  - `rmdata_old_snapshot`
- تأكد أن الأساسي يحتوي دمج الطرفين.

## 4.2 تحقق نوعي (عينات)
- افحص عينات من:
  - موظف تم تعديله على القديم.
  - موظف تم تعديله على VPS.
  - سجل لوج حديث.
  - مستند/صورة جديدة.

## 4.3 تحقق وظيفي من التطبيق

- [ ] شاشة الموظفين
- [ ] فتح الصور
- [ ] فتح المستندات

## خطة رجوع (Rollback)

في حال ظهور مشكلة حرجة:
- أوقف الوصول مؤقتا.
- استعد Backup `rmdata_db` قبل الدمج.
- استعد Backup الملفات قبل الدمج.
- عالج سبب المشكلة ثم أعد تنفيذ الدمج على بيئة اختبار إن أمكن.

---

## ملاحظات تنفيذية مهمة

- لا تستخدم استبدال كامل للقاعدة في هذه الحالة (لأن الطرفين فيهما إضافات).
- لا تعتمد `INSERT IGNORE` وحده للجداول الرئيسية لأنه لا ينقل "التعديلات" على السجلات الموجودة.
- لا تُنهِ النافذة قبل اختبار الصور والمستندات فعليا من داخل التطبيق.

## ملحق تشغيلي: المرحلة A (0.6 Bridge)

> هذا الملحق يطبق أثناء النقل التدريجي بين PHP وNode على VPS بدون توقف الخدمة.

## A.1 مراقبة أساسية أثناء الـ Cutover

- PM2 (Node):
  - `pm2 status`
  - `pm2 logs rmdata-node-api --lines 200`
- Nginx:
  - `sudo tail -f /var/log/nginx/access.log`
  - `sudo tail -f /var/log/nginx/error.log`
- Smoke checks بعد أي تعديل توجيه:
  - `curl -fsS https://api.rmdata.tech/api/health`
  - `curl -fsS https://api.rmdata.tech/node-api/health`

## A.2 إشارات التراجع (Rollback Triggers)

نفذ rollback فوري للـ endpoint المنقول إذا ظهر أي مما يلي:

- زيادة ملحوظة في `5xx` على المسار بعد التحويل.
- أخطاء SQL متكررة أو mismatch أعمدة/جداول.
- فشل login أو اختلاف `response shape` يكسر Electron IPC.
- أخطاء ملفات (`upload/open/delete`) أو فشل صلاحيات الوصول.

## A.3 خطوات Rollback سريعة (Endpoint-by-Endpoint)

1. أوقف canary للـ endpoint المتأثر (أعد التوجيه إلى PHP).
2. أعد تحميل Nginx:
   - `sudo nginx -t && sudo systemctl reload nginx`
3. تحقق فوري بعد التراجع:
   - Login
   - `db/query`
   - `files/list` (و `files/upload` إن كان المسار متأثرا)
4. راجع logs لتوثيق السبب قبل إعادة أي محاولة نقل.

## A.4 قاعدة الأمان في 0.6

- `PHP` هو المسار الافتراضي لأي endpoint غير مثبت parity.
- `Node` يستخدم كنطاق canary فقط حتى إغلاق فجوات 0.7.
- المرجع الرسمي لملكية المسارات:
  - [`docs/phase06_endpoint_inventory.md`](phase06_endpoint_inventory.md)
